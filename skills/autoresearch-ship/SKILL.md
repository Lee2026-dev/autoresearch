# autoresearch-ship

Creates a pull request for a completed task. You do NOT merge. You do NOT push to the default branch. You create the PR and stop.

---

## Context Received

- Task title and AC list from `prd.json`
- `gate_results.json` summary
- `review.md` (must show `Overall: APPROVED`)
- USR-ID and iteration number

---

## Procedure

### STEP 1 — Verify preconditions

Confirm all of these before proceeding:
1. `gate_results.json` has `"all_passed": true`
2. `review.md` has `Overall: APPROVED`
3. Working tree is clean: `git status`

If any precondition fails → **stop and report to orchestrator**. Do not create a PR with failing gates or unapproved review.

### STEP 2 — Ensure feature branch

```bash
git branch --show-current
```

If on `main` / `master` → stop and alert orchestrator. Coder should have worked on a feature branch.

If no feature branch was created, create one:
```bash
git checkout -b feat/<USR-ID>-<slug>
```

### STEP 3 — Push

```bash
git push -u origin HEAD
```

### STEP 4 — Build PR body

```markdown
## <Task Title>

**Task:** <USR-ID> | **Source:** <source field>

### Acceptance Criteria
- [x] <AC 1>
- [x] <AC 2>

All AC items verified by automated tests and independent reviewer.

### Quality Gates
| Gate | Status |
|---|---|
| format | ✅ |
| lint | ✅ |
| typecheck | ✅ |
| test | ✅ |
| security | ✅ |

<Reviewer positive observations if present>

---
🤖 Generated by autoresearch
```

If source is `github:<N>`, append `Closes #<N>` to the body.

### STEP 5 — Create PR

```bash
gh pr create \
  --title "<USR-ID>: <task title>" \
  --body "<PR body>" \
  --base main
```

Capture and return the PR URL to the orchestrator.

### STEP 6 — Report

```
[autoresearch:ship] PR created: <URL>
Waiting for human review and merge.
```

---

## Hard Rules

- **NEVER run `gh pr merge`** or any variant.
- **NEVER run `git push --force`**.
- **NEVER push directly to `main` or `master`**.
- If any precondition in STEP 1 fails, stop immediately. Do not proceed.
